System Architecture

Understanding the technical foundation of Ezchain's revolutionary blockchain design

Scale-Out Architecture

Ezchain introduces a novel scale-out architecture that breaks free from the limitations of traditional blockchain systems. The core innovation lies in its ability to distribute the workload across multiple specialized nodes, enabling linear scalability.

Node Types and Roles

Mining Nodes

Responsible for creating new blocks through proof-of-work mining. These nodes compete to solve cryptographic puzzles and add new blocks to the chain.

Specifications:

  • High CPU/GPU requirements
  • Participate in consensus
  • Earn block rewards

Storage Nodes

Specialized nodes that store the complete blockchain history and provide data availability to the network.

Specifications:

  • Large storage capacity
  • Full blockchain storage
  • Data retrieval services

Validation Nodes

Nodes that verify transactions and blocks, ensuring the integrity of the blockchain without participating in mining.

Specifications:

  • Moderate hardware requirements
  • Transaction verification
  • Block validation

Relay Nodes

Network nodes that facilitate communication between different parts of the network, optimizing data flow.

Specifications:

  • High bandwidth
  • Network optimization
  • Message routing

Light Client Nodes

Lightweight nodes that allow users to interact with the blockchain without storing the full chain.

Specifications:

  • Minimal hardware requirements
  • SPV verification
  • Mobile-friendly

Consensus Mechanism

Hybrid Consensus Algorithm

Ezchain employs a hybrid consensus mechanism that combines the best aspects of Proof-of-Work (PoW) and Practical Byzantine Fault Tolerance (PBFT) to achieve both security and scalability.

Fast Finality

Transactions are confirmed within one minute through optimistic execution

Energy Efficient

Optimized mining algorithm reduces energy consumption

Asynchronous Processing

Parallel transaction processing enables higher throughput

Key Data Structures

Block Structure

Each block contains transactions, a Merkle tree root, timestamp, nonce, and digital signature.

Block {
    index: int
    hash: str
    pre_hash: str
    miner: str
    time: datetime
    nonce: int
    m_tree_root: str
    bloom: list
    signature: str
}

Transaction Format

Transactions include sender, recipient, amount, nonce, signature, and timestamp.

Transaction {
    sender: str
    recipient: str
    nonce: int
    signature: str
    value: Value
    tx_hash: int
    time: datetime
}

Merkle Tree

Efficient data structure for verifying transaction inclusion in blocks.

MerkleTree {
    root: str
    leaves: list
    levels: list
}
